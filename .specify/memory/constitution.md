# forge Constitution

forge is a personal developer productivity CLI for macOS, written in Go. It will grow over time
to cover multiple domains (Docker management, project scaffolding, tmux orchestration, Git
workflows, etc.). The first domain is local Docker container management for per-project databases.

## Core Principles

### I. Modular Architecture (NON-NEGOTIABLE)

Every feature area is a self-contained module under `internal/modules/<name>`. A module owns its
own command tree, services, and templates. Modules MUST register themselves via a central registry
— `main.go` MUST NOT import individual modules directly. Adding a new module MUST NOT require
changes to existing modules.

### II. Single Static Binary

The CLI ships as one statically-linked Go binary installable via Homebrew. No runtime dependencies
on the host beyond Docker Engine itself for the docker module. Cold start MUST stay under 100 ms
for `--help` and listing commands.

### III. Docker as Source of Truth

For the docker module, the running Docker daemon is the source of truth — not local state files.
Managed resources are identified exclusively by labels:

- `forge.managed=true`
- `forge.project=<name>`
- `forge.engine=<name>`

The CLI MUST be able to fully reconstruct its view of the world by querying Docker labels alone.
Local state files, if any, are caches — never authoritative.

### IV. Engine Pluggability

Database engines (postgres, mssql, mysql, redis, mongo, …) implement a single `Engine` interface
and self-register via `init()`. Adding a new engine MUST require only one new file and zero changes
to command handlers or the Docker client wrapper.

### V. Predictable Naming & Idempotence

Every managed resource follows a deterministic naming scheme:

- Container: `forge-<project>-<engine>`
- Volume:    `forge-<project>-<engine>-data`
- Network:   `forge-net` (shared)

Every destructive command (`reset`, `remove`) MUST be idempotent — running it twice MUST NOT error
if the resource is already gone. Every destructive command MUST prompt for confirmation unless
`--yes` / `-y` is passed.

### VI. Safety Boundaries

The CLI MUST NEVER touch containers, volumes, or networks that lack the `forge.managed=true`
label. `list`, `remove --all`, and any bulk operation MUST filter by this label first. There is no
flag, no environment variable, and no config option that disables this check.

### VII. Spec-First, Atomic Tasks

All non-trivial work flows through Spec Kit: `/specify` → `/plan` → `/tasks` → `/implement`. Tasks
are atomic, independently committable, and reviewed individually. No multi-feature commits.

## Technology Constraints

- **Language**: Go 1.23+
- **CLI framework**: spf13/cobra + spf13/viper
- **Docker SDK**: github.com/docker/docker/client (official)
- **Terminal UX**: AlecAivazis/survey/v2 (prompts), fatih/color (output), olekukonko/tablewriter
  (tables)
- **Config location**: `~/.forge/config.yaml`
- **Target platform**: macOS (Apple Silicon primary, Intel supported); Linux nice-to-have

## Development Workflow

- Every feature starts with `/specify`. No code is written without an approved spec and plan.
- Tasks generated by `/tasks` are committed individually with conventional commit messages
  (`feat(docker): …`, `fix(core): …`).
- Each task that adds a public command MUST include a Hurl-style or shell-based smoke test under
  `test/smoke/`.
- `go vet ./...` and `gofmt -l` MUST pass before any commit.

## Governance

This constitution supersedes all other practices, guidelines, and conventions for the forge
project. Principles marked **NON-NEGOTIABLE** (currently: Principle I) MUST NOT be violated under
any circumstance and cannot be amended without a full review and re-ratification.

**Amendment procedure**:
1. Open a dedicated spec with `/specify` describing the proposed change and its rationale.
2. Increment the version according to semantic rules (MAJOR for breaking governance changes, MINOR
   for new principles or material expansions, PATCH for clarifications and wording fixes).
3. Update `Last Amended` to the date of the change.
4. Propagate any impact to dependent templates as described in the Sync Impact Report above.

**Compliance**: Every pull request MUST be verified against the Constitution Check gate in
`plan-template.md` before implementation begins. Complexity violations MUST be documented in the
Complexity Tracking table of the feature plan.
